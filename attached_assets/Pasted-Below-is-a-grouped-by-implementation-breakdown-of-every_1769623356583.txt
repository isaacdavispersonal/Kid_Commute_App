# Group 1 — Scheduling + Assignments + “Today” Resolution

## 1) Route assignment changes don’t reflect on the schedule (stale old route)

**Symptom**

* Admin changes a driver’s route assignment
* Schedule still shows the previous route

**Likely root causes**

* Schedule view is pulling from a **cached** source (TanStack Query key not invalidated)
* Assignment change endpoint updates one table, but schedule reads from a different source of truth
* The schedule is based on **Route templates** while assignment updates affect **RouteRuns** (or vice versa)
* Missing realtime emit: clients never receive “assignment.updated”

**Fix plan**

1. Identify source of truth:

   * Assignments should reference `driver_id + route_id + service_date + shift_type`
2. On assignment update:

   * Update DB record(s)
   * Emit realtime event: `assignment.updated` to `user:<driverId>` and `org:<orgId>`
3. Frontend:

   * Invalidate queries:

     * `schedule:today`, `driver:todayRoutes`, `admin:assignments`
   * If using socket events, update local store immediately (optimistic refresh)

**Acceptance checks**

* Change assignment → schedule updates without refresh within 2s
* Driver “My Routes” updates without closing app
* No old route remains after a hard refresh either (confirms backend consistency)

## 2) Driver “My Routes” shows only the first assigned route

**Symptom**

* Driver has multiple routes today
* UI only displays the first route

**Likely root causes**

* API returns array but UI uses `[0]` or `.find()` instead of mapping
* Backend query includes `LIMIT 1`
* Frontend grouping logic collapses multiple runs into one

**Fix plan**

* Backend: ensure endpoint returns **all** assigned routes for today (AM + PM)
* Frontend: render list; clearly label `AM/PM`
* Add empty state: “No routes assigned today”

**Acceptance checks**

* Driver with 2+ routes sees all routes
* Routes sorted by start time / shift
* Selecting each route loads correct stops/students

# Group 2 — Driver Route Execution Flow + Attendance + Stop Completion

## 3) Route management screen is inverted (students at top, stops below; card expands to complete route)

**What you want (authoritative behavior)**

* Students are tied to the **route stop**
* Flow should be: **Arrive stop → mark students → complete stop → next stop**
* Completing the **route** should be separate and only possible after all stops done (or with confirmation + pending review)

**Likely root causes**

* UI is built around a “global student list” not stop-scoped roster
* Stop completion logic isn’t enforcing “attendance marked”

**Fix plan**

1. Data:

   * Each student must have a stop association for pickup/dropoff (for that route/run)
2. UI:

   * Primary view = **current stop card**
   * Inside it: list of students assigned to that stop
   * Only allow “Complete Stop” if:

     * all non-absent students are marked
3. Route completion:

   * Only after last stop is complete, OR driver confirms “end early” → sends to Pending Review

**Acceptance checks**

* Arriving at stop unlocks attendance
* Stop cannot complete until attendance done (except grayed-out absents)
* Completing route triggers summary, not a hard lock

# Group 4 — Messaging, Notifications, Push, Badges, Notification Cards

## 9) Messages send but no in-app notification and no push

**Symptom**

* Message was sent
* Recipient gets neither in-app notification nor push

**Likely root causes**

* Message writes to DB but no “notify” pipeline triggers
* Device tokens not registered / not associated with user
* Push payload missing / failing silently
* Socket event not emitted or recipient not subscribed to correct room
* Notification permissions/token refresh not handled on iOS

**Fix plan (minimum)**

1. On message create:

   * Write message to DB
   * Create `notification` record for recipient
   * Emit socket event `message.created` to `user:<recipientId>`
   * Attempt push:

     * fetch all device tokens for recipient
     * send via FCM
     * log per-token response, revoke invalid tokens
2. Client:

   * ensure FCM token registration after login
   * ensure notification router handles deep links to thread

**Acceptance checks**

* Send message while recipient app closed → push arrives
* Open app → unread badge increments
* Tap push → opens correct thread
