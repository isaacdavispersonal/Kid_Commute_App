That log is actually super helpful. The **real blocker** is this:

**Minified React error #310** → the full error is: **“Rendered more hooks than during the previous render.”** ([React][1])

That means **some component is calling hooks (`useState`, `useEffect`, `useMemo`, etc.) conditionally** (inside an `if`, loop, or after an early `return`) so the number/order of hooks changes between renders. In production builds (like your Capacitor bundle), React throws this and you get a blank screen.

The iOS warnings (`UIScene lifecycle…`, Firebase swizzling, sandbox extension) are **noise** here — they’re not why your UI went blank.

---

## What usually causes this in your kind of app

Given your logs show `MobileAuth getAuthToken called, isNative: true` and then immediately React #310, the bug is likely in one of these places:

* `App.tsx` or your root router: “if token exists render X else render Y” **with hooks after the conditional**
* An auth gate component where hooks are only called when `isNative === true`
* A component that returns early when `loading` or `token` is missing, but hooks are defined after that return
* A screen that only exists in native builds and was recently edited

## How to fix “Rendered more hooks than during the previous render”

### ✅ Rule: hooks must run in the same order on every render

Bad (common pattern):

```ts
function MyScreen() {
  if (!token) return <Login />;   // ❌ early return before hooks

  const [x, setX] = useState(0);  // hooks called only sometimes
  useEffect(() => {}, []);
  ...
}
```

Good:

```ts
function MyScreen() {
  const [x, setX] = useState(0);
  useEffect(() => {}, []);

  if (!token) return <Login />;   // ✅ return AFTER hooks
  ...
}
```

Bad:

```ts
if (isNative) {
  useEffect(() => { ... }, []);   // ❌ hook inside condition
}
```

Good:

```ts
useEffect(() => {
  if (!isNative) return;
  // native-only behavior
}, [isNative]);
```
